# Freight Broker 사용자 가이드

비개발자·운영자를 위한 실행 방법, 일상 운영, 장애 시 확인 사항을 정리했습니다.

---

## 1. 시스템 구성

- **프론트(화면)**: 브라우저에서 접속하는 웹 주소 (예: `http://192.168.111.137` 또는 `https://...`)
- **백엔드(API)**: 데이터·문서·이메일 등을 처리하는 서버 (내부 포트 8000)
- **DB**: 주문·파트너·설정 등이 저장되는 데이터베이스 (PostgreSQL)

서버 한 대에 모두 올려 두고, Nginx가 80/443 포트로 받아서 프론트(5173)와 API(8000)로 나눠 줍니다.

---

## 2. 실행 방법 (서버에서)

개발 PC가 아니라 **Ubuntu 서버에서** 서비스를 띄우는 경우 보통 다음처럼 실행합니다.

1. **DB가 이미 떠 있는지 확인**  
   - Docker 사용 시: `docker ps` 에 DB 컨테이너가 있어야 합니다.  
   - 로컬 PostgreSQL 사용 시: `sudo systemctl status postgresql` 로 서비스 확인.

2. **백엔드(API) 실행**  
   - 프로젝트 폴더의 `backend` 안에서  
     `pip install -r requirements.txt` 후  
     `uvicorn app.main:app --host 0.0.0.0 --port 8000`  
   - 또는 배포 스크립트(`deploy_to_ubuntu.py` 등)에서 정한 방식대로 실행.

3. **프론트 실행**  
   - `frontend` 폴더에서 `npm run build` 후,  
     빌드 결과물을 Nginx가 제공하거나,  
     개발 모드라면 `npm run dev` (포트 5173).

4. **Nginx**  
   - 이미 설정되어 있다면 `sudo systemctl status nginx` 로 동작 확인.  
   - 설정 변경 후에는 `sudo nginx -t && sudo systemctl reload nginx`.

접속 주소는 Nginx에 설정된 대로 (예: `http://서버IP` 또는 `https://도메인`) 사용하면 됩니다.

## 3. 일상 운영

### 로그인

- 관리자가 알려준 **이메일**과 **비밀번호**로 로그인합니다.
- 역할(dispatcher, sales, billing, **admin**)에 따라 메뉴가 달라집니다.

### 메뉴별 용도 (요약)

| 메뉴 | 설명 |
|------|------|
| Dashboard | 요약 현황 |
| Order | 로드(화물) 목록·생성·수정 |
| Partner | 고객·배차처 등 파트너 관리 |
| Profit | 수익/비용 요약 |
| Invoicing | 매출(AR)·매입(AP) 인보이스 |
| Reports | 보고서 |
| Account | 사용자(계정) 관리 – **admin만** 표시 |
| Inventory | 재고 |
| Setting | 회사 정보·기본값·이메일(SMTP) – **admin만** 표시 |

### 문서·PDF

- Order(로드) 상세에서 **Load Confirmation**, **Rate Confirmation**, **BOL**, **Pallet Tag** 를 화면으로 보거나 **PDF**로 다운로드할 수 있습니다.
- Invoicing에서 고객 인보이스· carrier 인보이스도 **View** / **PDF** 로 확인·다운로드 가능합니다.

### 이메일 테스트 (admin)

- **Setting** → Email (SMTP) 섹션에서 받을 이메일 주소를 넣고 **테스트 메일 발송**을 누르면, 현재 설정된 SMTP로 테스트 메일이 발송됩니다.  
  실패 시 메시지를 확인하고, 서버의 `.env` 에 `SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`, `SMTP_PASSWORD`, `FROM_EMAIL` 이 올바른지 점검합니다.

## 4. 백업

- **DB 백업**: `deploy/backup_db.sh`  
  - cron 예: 매일 새벽 2시  
    `0 2 * * * /home/john/freight-broker/deploy/backup_db.sh`  
  - 백업 파일은 `BACKUP_DIR`(기본 `/home/john/backups`)에 `freight_broker_날짜시간.sql.gz` 형태로 쌓입니다. 30일 지난 파일은 스크립트에서 자동 삭제됩니다.

- **업로드 파일 백업**: `deploy/backup_uploads.sh`  
  - `backend/uploads` 를 tar.gz 로 백업하고, 30일 지난 백업은 삭제합니다.  
  - **정기 실행**: `deploy/setup_cron.md` 에 따라 cron 에 등록하면 매일 자동 백업·로그 정리됩니다.

---

## 5. HTTPS

- 도메인이 있으면 **Let’s Encrypt** 로 무료 인증서 사용을 권장합니다.  
  자세한 절차는 `deploy/setup_https.md` 를 참고하세요.
- 도메인 없이 테스트만 할 때는 같은 문서의 **자체 서명 인증서** 절을 참고하면 됩니다.

## 6. 장애 시 확인 사항

| 증상 | 확인할 것 |
|------|------------|
| 로그인 안 됨 | 이메일·비밀번호, DB/API 서버 동작 여부 |
| 화면이 안 뜸 | 브라우저 주소, Nginx 동작, 프론트(5173)·백엔드(8000) 프로세스 |
| 문서/PDF 안 나옴 | 백엔드 로그 확인. PDF는 weasyprint 설치·의존성 필요 (서버에 cairo 등) |
| 이메일 안 나감 | Setting의 테스트 메일로 확인. `.env` 의 SMTP 설정, 방화벽/587 포트 |
| 403 (권한 없음) | 해당 메뉴는 admin 전용일 수 있음. 계정 역할 확인 |
| 503 (PDF 불가) | weasyprint 미설치 또는 라이브러리 누락. 백엔드 로그 확인 |

서버 로그는 백엔드를 실행한 터미널 또는 systemd/supervisor 로그에서 확인합니다.

## 7. 모의 운영 테스트 (테스트 데이터 + 자동 검증)

배포 전·후에 **테스트 데이터를 넣고 API를 순서대로 호출**해 동작을 검증할 수 있습니다.

**Ubuntu 서버에서 할 일 요약**

1. **테스트 데이터 시드**  
   - `cd ~/freight-broker/backend`  
   - `source .venv/bin/activate` (가상환경 활성화 필수)  
   - `export DATABASE_URL="postgresql://postgres:postgres@localhost:5433/freight_broker"`  
   - `export PYTHONPATH=.`  
   - `python scripts/seed_test_data.py`  
   - → admin@test.com / dispatcher@test.com (비밀번호: test1234), 파트너·로드·인보이스 생성.

2. **모의 운영 스크립트**  
   - 같은 폴더·가상환경에서: `python scripts/run_mock_operation.py`  
   - → 로그인·대시보드·로드·LC PDF·인보이스 PDF·권한(403)까지 자동 검증.

**자세한 명령어·예상 출력·오류 해결**은 **`backend/scripts/README.md`** (모의 운영 테스트 가이드)를 보세요.

## 8. 나중에 할 수 있는 것 (선택)

| 항목 | 왜 하는지 |
|------|------------|
| **Nginx 리버스 프록시** | 브라우저는 80번 포트 하나만 쓰고, Nginx가 내부적으로 8000(API)·5173(프론트)로 나눠 줌. 포트 번호 없이 `http://서버IP` 만으로 접속 가능. |
| **프론트 빌드 + Nginx로 서빙** | 지금은 `npm run dev`(개발 서버). 배포 시엔 `npm run build` 로 만든 파일을 Nginx가 제공하면 더 가벼우며, 5173 프로세스를 안 켜도 됨. |
| **HTTPS** | 주소를 `https://` 로 쓰면 통신이 암호화됨. 도메인이 있으면 Let's Encrypt로 무료 인증서 가능. (`deploy/setup_https.md`) |

지금처럼 8000·5173 포트로 직접 접속해도 동작에는 문제 없습니다. 위는 “한 주소로 접속”, “속도/안정성”, “보안”을 더 챙기고 싶을 때 하면 됩니다.

---

## 9. 요약

- **실행**: DB → 백엔드(8000) → 프론트(빌드 또는 5173) → Nginx(80/443).  
- **운영**: 로그인 후 역할에 따라 Order, Partner, Invoicing, Reports 등 사용. Account·Setting은 admin만.  
- **백업·cron**: `deploy/setup_cron.md` 참고해 DB·업로드 백업·로그 정리 cron 등록.  
- **모의 운영**: `scripts/seed_test_data.py` → 서버 기동 → `scripts/run_mock_operation.py` 로 PDF 포함 전체 플로우 검증.  
- **문제 시**: 로그인/화면/PDF/이메일/403·503 항목 위 표대로 단계적으로 확인.

추가 문의는 개발자 또는 서버 관리자에게 요청하세요.
